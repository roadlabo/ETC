# UIお知らせ機能マニュアル

## 1. この機能は何か
ETC Analyzer の各UIを起動したときに、道路ラボからの新着お知らせを自動で表示する仕組みです。

たとえば、次のようなお知らせを出せます。

- 不具合修正のお知らせ
- 新機能追加のお知らせ
- 軽い操作注意点
- 会員向け詳細記事への案内

この機能の目的は、ソフト利用者が、UIを起動したときに自然に最新情報へ気づけるようにすることです。

---

## 2. どういう仕組みか
全体の流れは、次のとおりです。

1. WordPress 側で「UIお知らせ」記事を公開する
2. 各UIが起動時に、その一覧を取りに行く
3. まだ見ていない記事があれば、ダイアログで表示する
4. 「記事へ」を押した記事だけ既読にする
5. 「後で見る」を押した記事は、次回また出す

つまり、
**サイト側で配信し、PC側で既読管理する**
という構成です。

---

## 3. なぜこの方式にしたのか
### 3-1. 会員制サイトを壊さないため
etc.roadlabo.com は会員制の記事があります。
しかし、UI起動時のお知らせは、ログインなしで読めた方が使いやすいです。

そのため、記事を2種類に分ける方針にしました。

- **お知らせ版**
  - 一般公開
  - noindex
  - 内容は薄く書く
  - UIから直接開く
- **詳細版**
  - 会員限定
  - 修正内容や手順を詳しく書く

これにより、
**UI利用者はすぐ見れる**
一方で、
**詳しい内容は会員記事に逃がせる**
ようになります。

### 3-2. src 更新で既読情報を消さないため
ソフトは src フォルダを更新で差し替えることがあります。
そのため、既読管理ファイルを src に置くと、更新のたびに消えるおそれがあります。

そこで、既読情報は次の場所に保存することにしました。

`userdata/news/news_state.json`

これなら、ソフト本体更新と切り離して管理できます。

---

## 4. WordPress 側の構成
### 4-1. カテゴリ
UI通知専用の記事カテゴリを作成しています。

- カテゴリ名: ETCアナライザーUIお知らせ
- スラッグ: `ui-news`
- カテゴリID: `16`

### 4-2. 配信方法
WordPress REST API を使って記事一覧を取得します。

取得URL:
`https://etc.roadlabo.com/wp-json/wp/v2/posts?categories=16`

このAPIから主に次の項目を使います。

- `id`
- `title.rendered`
- `link`
- `modified`

### 4-3. 公開方針
UIから直接開く記事は、次の設定にします。

- 公開
- noindex
- Restrict access は OFF
- 内容は薄く書く
- 詳しい内容は会員向け記事へ誘導する

### 4-4. noindex の意味
noindex は、検索エンジンに「インデックスしないでください」と伝える設定です。
ただし、完全な秘匿ではありません。
URLを知っていれば見える可能性はあります。

そのため、公開記事には機密性の高い内容を書かず、
**軽いお知らせだけ**
を書く運用にしています。

---

## 5. Python 側の構成
UIお知らせ機能は、共通モジュールとしてまとめています。

### 5-1. ファイル構成
`src/common/news/`

その中に主に次のファイルがあります。

- `news_fetcher.py`
- `news_dialog.py`

### 5-2. 役割分担
#### news_fetcher.py
担当:
- WordPress API から記事一覧を取得
- 未読記事を抽出
- 既読情報の読み書き
- 既読キーの生成

#### news_dialog.py
担当:
- PyQt6 のダイアログ表示
- 「記事へ」「後で見る」ボタン処理
- 未読記事を順番に表示

---

## 6. 既読管理の仕組み
### 6-1. 既読ファイルの場所
`userdata/news/news_state.json`

### 6-2. 中身
例:

```json
{
  "seen_keys": [
    "2443:2026-03-21T10:39:43"
  ]
}
```

### 6-3. 既読キー
既読判定は、次の形で行います。

`記事ID:更新日時`

例:
`2443:2026-03-21T10:39:43`

### 6-4. この方式のメリット
同じ記事でも、更新日時が変われば別キーになります。
つまり、

- 新記事 → 通知される
- 既存記事の更新 → 再通知される

という動きになります。

---

## 7. ダイアログの動き
### 7-1. 表示内容
ダイアログには主に次が表示されます。

- ヘッダー
- 記事タイトル
- 更新日時
- 説明文
- ボタン2つ

### 7-2. ボタンの意味
#### 記事へ
- ブラウザで記事を開く
- その記事を既読登録する

#### 後で見る
- 今回は閉じる
- 既読登録しない
- 次回起動時にまた表示する

### 7-3. × ボタン
右上の × で閉じた場合も、
**後で見ると同じ扱い**
で、未読のままです。

---

## 8. 複数記事がある場合
未読記事が複数ある場合は、
**1件ずつ順番に表示**
します。

現在の仕様では、
**更新日時の古い順**
に表示します。

つまり、

1. まず古い未読記事
2. 次に新しい未読記事

の順になります。

これは、「前に出したお知らせから順番に読める」ようにするためです。

---

## 9. オフライン時の挙動
インターネットに接続されていない場合や、サイトへ到達できない場合は、UIお知らせ取得に失敗します。

この場合は、

- エラーでソフト全体を止めない
- ダイアログも出さない
- UI本体はそのまま起動する

という設計です。

つまり、
**お知らせ機能は失敗しても本体を邪魔しない**
ようにしています。

## 10. 各UIへの組み込み方
### 10-1. 対象UI
現在、この機能は次のUIに横展開する想定です。

- 01_1stScr_UI
- 02_UI_existence_trip_counter
- 03_UI_base_zone_estimator
- 05_trip_viewer
- 11_crossroad_sampler
- 21_UI_point_trip_extractor
- 31_32_crossroad_performance_to_report
- 33_branch_check
- 43_UI_peak30min_od

### 10-2. 組み込み位置
基本は各UIで次の位置に入れます。

- `QApplication(sys.argv)` の直後
- メイン画面生成前

### 10-3. スキップオプション
他UIから別UIを呼び出す場合、起動時通知が二重表示されることがあります。
そのため、次のオプションを用意しています。

`--skip-news-check`

これが付いている場合は、お知らせダイアログを表示しません。

---

## 11. import エラー対策
embedded Python 環境では、`common` モジュールをそのまま読めないことがあります。

そのため、各UIでは `src` を `sys.path` に追加してから import します。

考え方は、
**Python に「src をルートとして見てください」と教える**
ためです。

これを入れないと、次のようなエラーが出ます。

`ModuleNotFoundError: No module named 'common'`

---

## 12. ロゴクリック機能との関係
UIお知らせ機能とは別に、右上ロゴクリックで道路ラボを開く機能も追加しています。

これは別の共通部品です。

`src/common/ui/logo_link.py`

役割:
- 右上ロゴをクリック可能にする
- `https://etc.roadlabo.com` を開く
- オフライン時は
  `インターネット接続がされていません。`
  のダイアログを出す

つまり、共通部品は大きく2種類あります。

- **news 系**: UIお知らせ
- **ui 系**: ロゴリンク

## 13. Git / GitHub 運用
### 13-1. GitHub に上げる対象
上げるべきもの:
- `src/common/news/news_fetcher.py`
- `src/common/news/news_dialog.py`
- 各UI側の組み込み修正
- `src/common/ui/logo_link.py`
- `.gitignore`

### 13-2. GitHub に上げないもの
上げないもの:
- `userdata/`
- `__pycache__/`
- `*.pyc`
- `runtime/`
- `logs/`

### 13-3. .gitignore の考え方
`userdata/` を除外している理由は、
PCごとに既読状態が違うからです。

もしこれをGit管理すると、
ユーザーごとの差分でリポジトリが汚れます。

## 14. 運用ルール
### 14-1. 記事を書くとき
公開の UIお知らせ記事には、次のような内容だけを書く。

- 何が変わったか
- どのUIが対象か
- 軽い影響
- 詳細記事への案内

### 14-2. 書かない方がよい内容
公開記事には、次のような内容はできるだけ書かない。

- 内部ロジックの詳細
- セキュリティ上まずい情報
- APIキー
- 詳細な脆弱性情報
- 会員限定で見せたい深い内容

### 14-3. 更新時
記事を更新すると、`modified` が変わります。
そのため、既に見た記事でも、更新後は再通知されます。

---

## 15. トラブル時の見方
### 15-1. ダイアログが出ない
確認すること:
1. WordPress 記事が `ui-news` カテゴリに入っているか
2. 公開状態になっているか
3. APIで取得できるか
4. その記事が既読になっていないか
5. `--skip-news-check` が付いていないか

### 15-2. import エラーが出る
確認すること:
1. `common.news.news_dialog` の import 文
2. `src` を `sys.path` に追加しているか
3. embedded Python で実行しているか

### 15-3. 既読が消える
確認すること:
1. `userdata/` を消していないか
2. 本体更新で `userdata` を巻き込んで削除していないか
3. `news_state.json` が壊れていないか

---

## 16. 初心者向けまとめ
この機能をひとことで言うと、

**「WordPress で配信したお知らせを、各UI起動時に自動で出す仕組み」**
です。

大事なポイントは次の4つです。

1. お知らせ記事は WordPress 側で管理する
2. 詳細は会員記事、UIから開くのは軽い公開記事にする
3. 既読情報は `userdata/news/news_state.json` に保存する
4. UI本体は、お知らせ取得に失敗しても止めない

---

## 17. 今後修正するときの基本方針
今後、仕様変更があるときは、なるべく **CODEX 指示で修正** するのがおすすめです。

理由:
- 変更履歴がきれいに残る
- 横展開しやすい
- 再現しやすい
- 自分が忘れても、指示文を読めば戻れる

特に次のような修正は、共通部品側だけで済むことが多いです。

- 表示順変更
- 文言変更
- ボタン追加
- ダイアログデザイン修正
- 既読ルール変更
